Commercetools:
Reverse Image Search Engine

<reverse_image_search_brief>

SUMMARY
- End-to-end ownership of a production reverse image search pipeline (Image Search API) at commercetools. Not generative AI, but same building blocks: embeddings, retrieval, scalable inference, API design, monitoring. Took the feature from early prototype through deployment, scaling, and monitoring in ~3 months. Goal: return visually similar product images given a query image.

PROBLEM & GOAL
- Return visually similar product images given a query image (reverse image search). Support indexing of customer product catalogs and low-latency query API so clients could build “search by image” experiences (e.g. snap a photo, find similar products).

TECHNICAL APPROACH
- Model: Pretrained VGG16 CNN in Keras to extract dense feature vectors (embeddings) from images. VGG16 achieves ~92.7% top-5 on ImageNet; embeddings capture semantic meaning beyond pixel similarity. Dimensionality is VGG16-typical (e.g. 4096-d or 512-d from final layers depending on which layer was used); we tuned dimensionality vs. latency and storage.
- Similarity: Cosine similarity between query embedding and indexed embeddings (not Euclidean—cosine measures correlation/direction, better for high‑dimensional features). Cosine computed in SQL as a linear combination of columns for retrieval.
- Inference: Model hosted on Google ML Engine for elastic scaling with indexing and query demand.

ARCHITECTURE & INFRASTRUCTURE
- Indexing: Asynchronous processing of customer catalogs via Celery workers. Each unique image URL was vectorized, embeddings stored in PostgreSQL with product metadata. Many-to-many product–image mapping stored in separate tables. Hundreds of thousands of images indexed per catalog.
- Retrieval: User submits image via HTTP → embedding generated → cosine similarity computed in SQL against stored vectors → top‑k closest matches returned as image URLs in JSON.
- Storage: PostgreSQL for embeddings and metadata (no dedicated vector DB). Exact cosine search in SQL was sufficient for our scale and kept ops simple.

SCALE, LATENCY, PERFORMANCE
- Latency target: responses within ~3 seconds end-to-end.
- Scale: Hundreds of thousands of images per customer catalog; indexing and query load handled by ML Engine scaling and Celery.
- Tradeoff: Tuned embedding dimensionality to keep retrieval quality while staying within latency and storage budgets.

TIMELINE & OWNERSHIP
- ~3 months from prototype to production. I was the main owner (junior at the time); one other SWE supported with DevOps and general guidance. I drove model choice, pipeline design, API, and integration; he helped with deployment and infrastructure.

VALIDATION & QUALITY
- Validation was hard: beyond training/embedding loss we had no automated metric for “visually similar” in production. We manually inspected image results to ensure the system was retrieving sensible matches. No formal offline metric (e.g. recall@k on labeled pairs); quality was judged by inspection and adoption.

TRADEOFFS & KEY DECISIONS
- Pretrained model vs. training our own: Shipped faster with strong baseline; gave up domain-specific optimization (e.g. e‑commerce‑tuned). Plan was to improve with more e‑commerce data and feedback.
- PostgreSQL vs. dedicated vector DB: In 2018, FAISS and production vector DBs were less mature. Staying in PostgreSQL reduced new infra and ops risk; exact cosine in SQL was fast enough at our scale.
- Embedding dimensionality: Balanced retrieval quality vs. latency and storage; tuned in practice.

CHALLENGES & LESSONS
- Validation: Relying on manual inspection is limiting; would want labeled similarity data and recall@k (or similar) for iteration.
- Scale and tooling: Exact search in SQL doesn’t scale to very large or very high‑QPS regimes; modern vector DBs or approximate search (e.g. FAISS) would be better at larger scale.

WHAT I’D DO DIFFERENTLY TODAY
- Use a dedicated vector DB or FAISS for indexing and retrieval at scale.
- Use a richer or more modern embedding model (e.g. CLIP, or domain‑fine‑tuned models) for better semantic similarity.
- Add proper offline evaluation (labeled pairs, recall@k, etc.) and monitoring of retrieval quality.

IMPACT & CUSTOMERS
- Success measured by adoption. Carhartt was a major customer. Use cases included similar-product discovery (e.g. photo → similar items), duplicate detection, and image authentication. API enabled demos and customer integrations (e.g. demo app from commercetools using the Image Search API).

TRANSFER TO GENAI / INTERVIEW TALKING POINTS
- Same core components as many GenAI systems: embedding generation, retrieval, scalable inference, API design, production monitoring. Experience generalizes to building and scaling GenAI platforms (RAG, semantic search, embedding pipelines). Can speak to: why cosine vs. Euclidean, why PostgreSQL at the time, pretrained vs. custom training, validation challenges, and tradeoffs between simplicity and scale.

</reverse_image_search_brief>

Grandperspective: 
Cloud Sync Backup System

<grandperspective_cloud_backup_etl_brief>

SUMMARY
- At Grandperspective GmbH: early-warning gas leak detection for industrial factories using IR optical measurements. I helped build data infrastructure (signal processing, data visualization, Google Cloud, ETLs) and worked closely with Data Science and DevOps on platform uptime, data availability, and end-to-end monitoring and testing. I owned the cloud backup system and ETL systems. Remote sensors were on-prem, often in deserts or locations without reliable internet. Devices produced one image and ~60–100 infrared spectra every 6 seconds → several GB per hour. Data was backed up in batches (streaming was tested): package Postgres data into .avro → upload to GCS for archival → Cloud Function initially dumped into BigQuery; later the second half was refactored to Apache Beam/Dataflow for aggregations, pluggable aggregation types, and analysis over months of data.

ARCHITECTURE (BACKUP + ETL)
- On-prem: Postgres (later TimescaleDB) with sensor data (images, IR spectra, metadata). Indexes on time and scan area; large columns (images, large JSON settings, IR spectra) lazy-loaded where possible.
- Backup path: Batch packaging of Postgres data into .avro files → upload to GCS (archival). Retries and checks for incomplete/finished uploads; backfilling scripts for longer outages when internet returned.
- Ingest to cloud: Initially Cloud Function dumped GCS data into BigQuery. Later refactored to Apache Beam/Dataflow for ETL: aggregations, pluggable aggregation types, support for viewing months of data.
- Scale: Several GB per hour per site; batches sized for reliability over poor links.

<schema_evolution_qa />
- We kept migrations backwards compatible: only adding nullable columns or removing columns (no breaking changes). We used Alembic for versioned migrations so we had rollbacks and clear traceability. Testing flow: SQLite first, then staging Postgres, then live Postgres. That gave us a safe path to evolve schema without breaking running ingest or analytics.

<late_arriving_data_qa />
- Late-arriving data was acceptable; we had no real-time use case. Real-time needs were served from the live on-prem Postgres, not the cloud pipeline. We did have retry mechanisms and alerting for missing or incomplete batches. The main cause of missing data was poor internet and failed uploads, so we used retries, checked for internet before starting uploads (fail early), and backfilling scripts to detect longer gaps and retry in batches when connectivity returned.

<idempotency_qa />
- BigQuery does not guarantee exactly-once ingestion; as a distributed system it can produce duplicates, and we had to handle that. We reduced duplicates by: (1) uploading on a fixed schedule (e.g. hourly) with each file uniquely named by hour and scan area so the same logical batch had one canonical file; (2) checking for completed uploads before attempting upload to avoid re-sending the same batch; (3) monitoring in our observability stack (e.g. Datadog) for missing and duplicate sections of data so we could tell “system down, no data expected” vs. “data genuinely missing.” Once BigQuery supported DELETE, we ran a daily job that enforced uniqueness on a composite key (time + scan area) and deleted duplicates.

<biggest_failure_challenges_wins_qa />
- Biggest frustration: unreliable internet at remote sites (deserts, no connectivity). We addressed it with retries, checking for incomplete vs. finished uploads before retrying, and backfilling scripts that detected longer failures and re-ran batches when internet was back.
- RAM pressure: Avro files were several GB. We avoided loading whole files into memory by appending one row at a time to Avro (append-only pattern) instead of reading the full file. We also added strategic indexes (time, scan area), lazy-loaded large columns (images, large JSON, IR spectra), and migrated from Postgres to TimescaleDB for better time-based indexing and queries.

<why_timescaledb_over_postgres />
- Our workload was time-series: high-volume sensor data keyed by time and scan area, with heavy time-range queries (e.g. “last 6 hours,” “last month”). TimescaleDB is Postgres extended for time-series: (1) Hypertables automatically partition by time, so time-range filters hit fewer chunks and indexes, improving query and retention management. (2) Time-oriented compression and retention policies reduce storage and keep hot recent data fast. (3) Time-bucketed aggregations (e.g. time_bucket) are native and efficient. (4) We already had time and scan area as primary access patterns; TimescaleDB’s partitioning and indexing aligned with that and gave better performance than plain Postgres on the same schema.

<if_i_redesigned_today />
- Streaming where it helps: Use streaming ingest (e.g. Dataflow with streaming, or Pub/Sub + Dataflow) for lower latency and smaller, more frequent commits where the network allows, while keeping batch/backfill for poor connectivity.
- Managed ingest and dedup: Use BigQuery streaming insert with insertId or batch load with idempotent keys; or use a write layer (e.g. Dataflow) that deduplicates on (time, scan area) before writing to BQ so we don’t rely on daily DELETE.
- Schema and evolution: Use a schema registry (e.g. Avro schema in a registry) and versioned schemas so downstream (Beam/Dataflow, BQ) can handle evolution and backfills consistently.
- Observability: Structured logging and metrics at each stage (export to GCS, load to BQ, Dataflow), with alerts on duplicate/missing windows and data freshness so we detect issues before backfill.
- Storage and format: Consider columnar formats (e.g. Parquet) in GCS for analytics and compression; keep Avro if we need row-by-row append or schema evolution in the file layer.
- TimescaleDB still makes sense on-prem for time-series; in the cloud we might evaluate BigQuery’s native time-partitioning and partition expiry as the single store for analytics instead of duplicating retention logic.

</grandperspective_cloud_backup_etl_brief>
